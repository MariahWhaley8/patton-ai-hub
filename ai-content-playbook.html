Playbook

Build a Slack AI Assistant Over Your Own Data

A step-by-step method for a Slack-native assistant that answers staff questions from your organization's own records and documents — and writes updates back. Hosted cheaply, accurate by design, easy to maintain.

~Half a day to build Managed, low-cost hosting Hybrid SQL + vector retrieval

Overview

What you're building

An assistant that lives in Slack and pulls from two stores you control.

Slack  ──►  Assistant (Claude, tool-using agent)
                │
                ├─ search_records    → direct read-only SQL over synced data   (exact facts)
                ├─ search_documents  → vector search over docs + notes         (context)
                ├─ find_record       → resolve a name to a record id
                ├─ update_record     → write changes back to the source system
                └─ create_record     → create a new record (only after user confirms)
        Postgres (with pgvector) stores both a structured mirror and the vectors.

The design is a deliberate hybrid:

  • Exact facts (statuses, amounts, dates, counts) come from a direct SQL query against a synced mirror of your data — deterministic, no "closest guess."
  • Narrative / documents use semantic search (embeddings + pgvector).
  • Only the data each question needs is pulled, and the static prompt is cached — so it's fast and cheap.

Build it

The steps

1 · Get API access to your data source

Most low-code/records platforms expose a REST API where each record type is an "entity":

GET/POST/PUT  https://<host>/api/apps/<appId>/entities/<Entity>
Header:       api_key: <YOUR_KEY>   # some use Authorization: Bearer — check the docs

Generate an API key, confirm which verbs work (GET/POST/PUT; some don't support PATCH), and test a simple GET before writing code.

Pitfall: store only the key value — pasting a whole docs snippet into the token variable produces a malformed auth header and 403s.

2 · Provision hosting + database

  • Create a Railway project and add Postgres with the pgvector extension.
  • Railway auto-provides DATABASE_URL.
  • Two stores: a structured mirror of your records (for SQL answers) and documents + chunks with a vector(1024) column and an HNSW index (for semantic search).
Pitfall: create the vector extension on a plain connection before registering the pgvector adapter — otherwise the client can't find the type on first boot.

3 · Model + embedding keys

Anthropic: create a key, set ANTHROPIC_API_KEY and a current model string. Voyage AI: create a key; use a domain-tuned model (e.g. voyage-law-2, 1024 dims) if it fits.

Pitfall: Voyage's free tier is rate-limited until you add a payment method — the large free token grant still applies, so bulk ingestion stays effectively free but no longer crawls.

4 · Create the Slack app

  1. Create the app (pasting an App Manifest with only the scopes/events below avoids "enterprise/admin-only" install errors).
  2. Socket Mode → On; generate an app-level token with connections:write (the xapp- token). No public URL needed.
  3. Bot Token Scopes: app_mentions:read, chat:write, im:history, im:read, im:write, channels:history, channels:read, groups:history, groups:read, reactions:write.
  4. Event Subscriptions → bot events: app_mention, message.im.
  5. App Home → Messages Tab On, and check "Allow users to send messages from the messages tab" (this is what lets people DM the bot).
  6. Install the app; copy the Bot User OAuth Token (xoxb-) and the Signing Secret.
  7. Set the app name, bot display name, and a 512–2000 px square PNG icon.
Two pitfalls that will bite you: after adding any scope later you must Reinstall to Workspace for the token to receive it; and the sidebar name comes from App Home → Display Name (renaming needs a reinstall + Slack reload).

5 · (Optional) Google Drive document source

  1. Enable the Drive API; create a service account and a JSON key.
  2. Put the whole JSON into GOOGLE_SERVICE_ACCOUNT_JSON.
  3. Share the target folder with the service account's email — it only sees folders shared with it.
  4. Set GDRIVE_FOLDER_ID to the bare id from the URL (after /folders/) — not the full URL.

6 · Build the app (key implementation notes)

Modules: config, db, embeddings, retrieval, tools, agent, slack_app, writeback, and one *_sync per source. The parts that are easy to get wrong:

  • Read-only SQL guard: allow only a single SELECT/WITH, reject write keywords, set a statement_timeout.
  • Cast query embeddings: send the vector as a text literal cast to ::vector — a raw list is sent as a float array and errors.
  • Sanitize text before storing/embedding (strip NUL/control chars — Postgres rejects 0x00).
  • Tool loop: call the model; on tool_use run tools, append results, loop (cap ~5–6). Prompt-cache the system prompt + tools.
  • Reactions: add 👀 on receipt, swap to ✅ on success, ⚠️ on error (best-effort).
  • Thread memory: pass prior thread messages so multi-turn flows work.
  • Write-back with a confirmation gate: updates apply immediately; creating a new record only after the user says yes — enforced in the prompt and in the function.
  • Audit trail: every write appends a timestamped note to the record and optionally posts to a log channel.

7 · Deploy

Push to GitHub → Deploy from GitHub repo on Railway (set the service Root Directory if code is in a subfolder) → add all environment variables → Railway auto-redeploys on every push.

8 · Load the data (ingestion)

python -m ingest.run_ingest all       # everything
python -m ingest.run_ingest records    # structured mirror (no embeddings — load first)
python -m ingest.run_ingest drive      # documents (incremental — skip already-indexed)
python -m ingest.run_ingest chat       # chat history

Make ingests idempotent and document ingests incremental; for long jobs run detached (nohup … &) so a disconnect can't kill them.

9 · Test

Test the agent directly (a fact question + a document question), then in Slack: invite the bot, @mention it, DM it, confirm the 👀 → ✅ reactions, and verify write-back (update applies; create asks first).

Think like a shareholder

What a shareholder actually cares about

Across the lifecycle — with rough time, money, and skill for each phase.

A shareholder isn't buying "an AI bot." They're buying return on a small, controllable investment: faster staff, fewer mistakes, a predictable run-rate, low risk. Frame every phase around ROI, risk, and cost predictability. Estimates assume a small team and one capable builder; treat dollar figures as directional and verify current provider pricing.

Don't repeat these

Pitfalls checklist

  • Token variable holds only the key — no extra text.
  • pgvector extension created before registering the vector adapter.
  • Query embeddings cast to ::vector in SQL.
  • Text sanitized (NUL/control chars) before insert/embed.
  • Per-file ingest wrapped in try/except so one bad file doesn't abort the batch.
  • Embedding provider payment method added to lift the free-tier rate limit.
  • Slack scope changes followed by a reinstall.
  • Folder id is the bare id, not the full URL.
  • On host AI/agent panels, don't type shell commands into an "agent" box — use the real Console/shell.

Security

  • Keep all secrets in host environment variables only.
  • Rotate any key ever pasted outside the host (screenshots, logs, chat).
  • The SQL tool must be read-only, single-statement; writes go only through explicit functions.
  • Limit which channels the bot is invited to — it can read and answer from their history.
  • Instruct the assistant to answer only from retrieved data, never invent facts, and flag anything needing professional review.
